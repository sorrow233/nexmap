1. 给后续排查提供一份“为什么这里会变成现在这样”的上下文。

3. 把“用户真实反馈 -> 根因判断 -> 代码改动 -> 回退教训”完整保留下来。

## 3. 这一阶段最重要的总体结论

### 3.1 真正反复被证明最危险的链路

最近这 100 多次提交里，最危险、反复爆炸、并且导致最多回退的，并不是单个 UI 组件，而是以下几条链路：

1. `live store -> 整板 snapshot -> Y.Doc / Firestore`
2. `metadata 同步 -> 缩略图资源上传 -> Firestore 写队列`
3. `流式 token -> cards.messages 持续落盘 -> Markdown / 滚动 / 持久化连锁更新`
4. `画布交互结束(mouseup / dblclick / keydown) -> 重计算 / 重绘 / 冗余 setState`

### 3.2 近期被反复修正的核心认识

这段提交历史说明团队在持续修正几个“看起来合理但实际很危险”的旧认识：

1. 不是所有同步问题都能靠调 debounce 数字解决。
2. 不是所有“远端更慢”的问题都应该立刻强推完整快照修复。
3. 不是所有流式体验优化都应该让内容实时落盘到正式消息结构。
4. 不是所有界面变化都适合在一个大组件里硬改，很多问题的真正根因是职责耦合。

### 3.3 回退为什么这么多

回退密集并不单纯说明开发“混乱”，更准确地说，它暴露了以下现实：

1. 同步层和性能层之间耦合太深，改 A 很容易放大 B。
2. 很多问题表象相似，例如：
   - Firestore `resource-exhausted`
   - 数据不同步
   - 打开大画布卡顿
   - 回答结束后消息消失
   实际根因却不一样。
3. 用户反馈往往是“线上真实高压场景”，一旦判断主因错了，错误修复会立刻暴露。

## 5. 问题簇深度分析

## 5.1 同步与 Firestore：最复杂、最贵、最容易误判的主线

### 5.1.1 主要症状

- `FirebaseError: [code=resource-exhausted]`
- `Write stream exhausted maximum allowed queued writes`
- 新设备外面显示卡片数量正确，进去内容缺失
- 超大画布一打开就卡、就写、就爆
- 本地空模板反向覆盖远端真实数据
- 单卡长对话无法主动同步

### 5.1.2 这条线真正踩过的坑

1. 把同步问题误判成“只要调大 checkpoint 分片、调慢 debounce 就行”。
2. 把远端修复触发绑在 `live snapshot` 或脏游标上，导致每次局部变化都重新桥接整板快照。
3. 把 metadata 同步与缩略图资源上传绑成一条主链。
4. 把“缺失卡片”误解释为“应该删除卡片”。
5. 把 revision 当成唯一真相，却没有给 revision 机制做完整性校验兜底。

### 5.1.3 这一阶段真正有效的修正

被事实证明更靠谱的方向：

1. 只在本地持久化成功后桥接远端同步，而不是从 live store 高频桥接。
2. metadata 与完整 board checkpoint 逻辑分离。
3. 缩略图资源上传独立排队，不能和 metadata 批量写同一条主链。
4. cards 同步按 `card.id` 合并，缺失默认保留，删除必须显式表达。
5. 引入完整性 hash 与低频 integrity monitor，给 revision 体系补慢速真相校验。

### 5.1.4 典型代表提交

- `610ee80b` / `c9364c14`：修复“外层数量对、内容不对”的 metadata / snapshot 双链问题。
- `1e987359`：移除打开画布就重型修远端的定时器。
- `ed831137`：禁止空本地快照回灌覆盖已有内容。
- `1cb82bd2`：卡片同步改为 Add-Wins 风格的显式删除语义。
- `3a255af6`：同步桥改为“本地持久化成功事件驱动”。
- `2c4f6e7f`：给 revision 体系补完整性校验。
- `a255d639`：缩略图资源上传改为独立串行队列。

### 5.1.5 这条线留下的最重要经验

1. 大画布同步问题绝大多数不是“某次提交突然变坏”，而是整板级放大链没被切断。
2. “看起来更实时”的方案，往往更容易把局部改动变成整板级写放大。
3. 一旦 metadata 与真实内容快照分离，就必须明确哪条链路负责展示，哪条链路负责恢复，哪条链路负责远端修补。

## 5.2 性能与卡顿：从“打开画布 5 秒”到“流式回答卡到结束”

### 5.2.1 主要症状

- 打开画布卡片要 `5s` 才出现。
- `dblclick / mouseup / keydown / requestAnimationFrame` 长任务高频出现。
- 打开 ChatModal 后主线程持续忙。
- AI 回答期间，从开始一直卡到回答结束。

### 5.2.2 被确认过的根因

1. 画布直接订阅整份 cards，导致内容变一下整板重算。
2. 增量索引名义上增量，实际上每次 cards 引用一变仍全量同步。
3. 连接线可见性、逐条 stroke、动画帧空转导致 rAF 时间爆炸。
4. 拖拽结束时同时做预览 flush、真实提交、清空预览。
5. 流式 token 每批都直接写回 cards.messages，导致 Markdown、滚动、持久化脏状态一起抖。

### 5.2.3 这一阶段最关键的性能改造

- `b908bb7b`：画布订阅从整份 cards 收缩到 `cardIndexMutation`。
- `90d85370` / `d735b4a7`：把 cardIndexMutation 继续轻量化，并区分 `content / geometry / bulk`。
- `e6027749`：暂停模态打开后背景连接层绘制，删掉无必要动画。
- `6240612d`：连接线按颜色合批绘制，且只在必要时重绘。
- `dc9c5176`：收紧 mouseup / keydown / click 末端的冗余更新。
- `64709828`：流式回答期间只写 streaming buffer，不再每批 token 都落盘到 cards。

### 5.2.4 性能线的关键教训

1. 性能问题很少只在“渲染层”，大多数是状态更新策略与副作用链设计问题。
2. 流式体验如果把“视觉即时性”和“正式状态持久化”混在一起，必然会卡。
3. 连接线和滚动这类“看起来只是视觉层”的逻辑，实际上最容易拖累主线程。

## 5.3 AI 与模型：并发、流式、排队、模型锁定语义连续重构

### 5.3.1 主要问题

- 同卡假并发，实际串行。
- 等待队列被意外移除，排队失效。
- 长回答截断，尤其是系统积分流式尾包。
- assistant 消息结束后消失或被吞掉。
- 默认模型与卡片锁定模型语义混淆。

### 5.3.2 关键修正

1. 增加持久化等待队列与排队 UI。
2. 修复尾包与中断时的 assistant 消息落盘。
3. 调整 Gemini 并发门闩和官方流式并发上限。
4. 将“卡片模型锁定”严格限定为用户显式选择行为。
5. 控制台记录真实模型调用，方便判断到底是哪个 provider / model 在跑。

### 5.3.3 这一线最重要的经验

1. “卡片记住模型”是强语义，不允许默认配置偷偷写死到卡片。
2. 流式消息临时态和正式消息态必须分开。
3. 等待队列不只是功能，它还会直接影响用户视角和滚动策略。

## 5.4 背景图、画廊展示与排序：展示层与正式快照脱节后的连续修正

这一组问题表面分散，实际上都指向一个核心：

`boardsList / metadata / 正式 board 快照 / 本地缓存` 四层数据之间的一致性没有被严格控制。

表现为：

- 背景图“掉了”，但正式快照其实还在。
- 自动背景生成重复触发。
- 主页排序与展示依据不一致。
- 不同设备看到的画廊顺序不一致。

关键修正包括：

- 从正式快照回填 backgroundImage / thumbnail / summary。
- 空值不允许覆盖真值。
- 主页排序专门拆出展示逻辑，不再直接复用同步顺序比较器。
- 最终统一回到“按创建时间排序并展示创建日期”的方案。

## 5.5 设置页、暗色模式与产品化界面收敛

最近这段历史里，设置页并不是简单换皮，而是经历了：

1. 密度太大、模块像营销页。
2. 夜间模式“像没适配”，因为关键容器仍是半透明浅层。
3. 主题信号混用了 runtime 与 Tailwind 的不同模式。

最终走向是：

- 设置页整体参考更“系统面板化”的语言重写。
- 暗色模式统一为明确深色面板，而不是灰白玻璃层。
- 全站 `html.dark` 与 Tailwind `class` 模式统一。

## 5.6 S3 备份：从“能上传”到“长期可用”

03-24 的 S3 备份链路是这一阶段少数明显向稳定功能收口的例子。

演进路径：

1. 先支持本地全量数据上传到用户 S3。
2. 再补列表与恢复闭环。
3. 再补完整性校验与本地 safety backup。
4. 再把保留策略收口为“最多 5 份”的简单稳健模型。
5. 再增强可诊断性。

这条线的特征是：

- 问题没有同步/性能那么爆炸。
- 但修复节奏明显更稳，方向也更聚焦。
- 是近期少数“由功能性做到可长期使用”的完整闭环。

### 6.1 被证明代价很高的错误方向

1. 在渲染层或 effect 层高频构造整板快照再桥接远端。
2. 用 metadata 去推断完整内容一定已经同步。
3. 让流式 token 实时落盘到正式 cards。
4. 在同步层给“旧但有 cards 的本地快照”留宽松 fallback。
5. 为了“减少日志噪音”直接隐藏关键诊断输出。

### 6.2 被证明正确但需要补护栏的方向

1. 缩略图资源从 metadata 拆出是对的，但必须单独排队。
2. 单卡模型持久化是对的，但必须只由用户显式锁定。
3. S3 全量备份是对的，但必须补校验与恢复前安全备份。
4. 画布增量索引是对的，但 mutation 信号必须足够轻且分层。

## 8. 对后续开发的建议

### 8.1 同步层

后续所有同步相关改动，必须优先回答下面 3 个问题：

1. 这次改动会不会让“局部变化被放大成整板写入”？
2. metadata 和正式快照的职责边界有没有变模糊？
3. 新设备空状态、旧快照、缺字段、缺卡这几种极端情况有没有被重新打开口子？

### 8.2 性能层

后续所有性能相关改动，必须优先回答下面 3 个问题：

1. 这个更新是写正式状态，还是只该写临时缓冲？
2. 这个 effect / 订阅 / setter 是不是在无变化时仍然白跑？
3. 这个动画或滚动逻辑是不是在模态打开、回答流中、拖拽结束时持续刷帧？

### 8.3 产品交互层

后续所有 UI 改动，建议遵守：

1. 不要把等待队列、设置页、模型切换器这类“有强语义”的组件只当成视觉问题。
2. 风格试验要小步，尤其是用户已经明确表达“只要这个模块，不要牵动其他地方”时。
3. 先验证行为语义正确，再压视觉层级。

## 10. 最后结论

如果只用一句话概括最近这 100 多次提交：

这不是一次普通的“功能迭代期”，而是一段围绕 `同步安全`、`性能稳定` 和 `状态语义` 的高压纠偏期。

1. 同步必须从“整板级放大”收口到“持久化成功后的明确桥接”。
2. 流式回答必须把“临时视觉态”和“正式持久态”彻底拆开。
3. 画布与聊天性能问题，最终都要落到“哪些状态真的该更新，哪些根本不该动”。

这三条，是最近 120 次提交里最重要的技术经验。
